## Required release gates

The same accepted source commit must pass:

- pinned provider contract verification for the exact 1.1.0 artifact set;
- clean formatting, `go vet ./...`, uncached unit tests, full race tests, and `go build ./cmd/studio`;
- protected Linux Go browser E2E covering semantic structure, keyboard paths, tenant isolation, conflict handling, and dangerous confirmations;
- dependency review, vulnerability scanning, and CodeQL;
- review of OIDC/CSRF, redirect, role, bounds, idempotency, optimistic version, redaction, secret, and provider-internal-data boundaries.

Protected Linux Chrome is the browser authority. Local Windows browser execution is supporting evidence only.

## Supported deployment

- One Go Studio process behind production TLS termination.
- HTTPS Studio, OIDC, redirect, and provider endpoints in production.
- `STUDIO_CONFIG` for non-secret tenant registration and environment-variable references for client secrets.
- Chrome matching the protected Ubuntu runner.
- Keyboard navigation and semantic HTML are required. No named WCAG conformance level is claimed without an independent audit.

## Known limitations

1. Sessions, OIDC transactions, tokens, provider clients, caches, drafts, and idempotency state are volatile. Restart logs users out; multi-instance deployment requires affinity and has no shared-session failover.
2. TLS termination, secret injection, process supervision, provider backups, and provider HTTP handler bindings belong to the deployment owner.
3. Studio is fixed to contract 1.1.0 at the pinned provider commit and does not negotiate arbitrary contract versions.
4. Only Chrome on protected Ubuntu CI is release-certified. Firefox and Safari are not certified for v1.0.0.
5. Metadata relationship behavior is limited to frozen contract fields; Studio does not infer relationships from names.
6. Usage and discrepancy pages are observational and are not synchronous quota-enforcement authority.
7. Audit details are intentionally not rendered or exported beyond bounded redacted fields and `redacted_paths`.
8. Studio exposes no passwords, tokens, certificates, raw SQL, workflow payload/results, worker identity, or fencing-token controls.
9. The release makes no reproducible binary or container-image claim.

## Rollback

1. Remove new traffic from the rejected Studio deployment and preserve provider/audit evidence.
2. Redeploy S5 commit `f02a750141b122703e57e6c715adf1c98c0141ee`.
3. Restart Studio and require every operator to authenticate again. Volatile sessions, tokens, OIDC exchanges, drafts, caches, and mutation keys are intentionally discarded.
4. Keep the provider at the contract-compatible pin; a provider or contract downgrade requires a separate compatibility decision.
5. Re-run health, contract verification, authentication, tenant switching, and one read-only operation in metadata, identity, workflow, quota, and audit before restoring traffic.

Rollback does not undo metadata applies, identity mutations, workflow commands, or quota publications accepted by the provider. Those operations remain provider-authoritative and audited and require explicit compensation.
